## 1. プロジェクト概要

### 1.1 背景・目的

営業担当者が日々の顧客訪問活動を報告し、上長がフィードバックを行うための日報システムを構築する。現状の紙やメールベースの報告を電子化し、以下を実現する。

- 営業活動の可視化と一元管理
- 上長による迅速なフィードバック（コメント機能）
- 訪問履歴の蓄積と検索

### 1.2 システム名

営業日報システム

### 1.3 対象ユーザー

| ユーザー種別 | 役職 | ロール | 主な利用目的 |
|------------|------|--------|-------------|
| 営業担当者 | 担当・主任 | 一般 | 日報の作成・編集・閲覧 |
| 上長 | 係長・課長・部長 | 上長（管理者） | 日報の閲覧・コメント、マスタ管理 |

---

## 2. システム構成

### 2.1 システム方式

Webアプリケーション（ブラウザベース）として構築する。

### 2.2 対象ブラウザ

| ブラウザ | バージョン | 優先度 |
|---------|-----------|--------|
| Google Chrome | 最新版 | 必須 |
| Microsoft Edge | 最新版 | 必須 |
| Safari | 最新版 | 推奨 |

### 2.3 アーキテクチャ概要

```
┌────────────┐       ┌────────────┐       ┌────────────┐
│  ブラウザ   │──────→│  APIサーバー │──────→│ データベース │
│（フロントエンド）│←──────│（バックエンド）│←──────│            │
└────────────┘  REST  └────────────┘       └────────────┘
                JSON
```

- フロントエンド: SPA構成
- バックエンド: REST API（JSON形式）
- 認証: JWT（Bearer Token）方式

## 3. 機能要件

### 3.1 機能一覧

| 機能ID | 機能名 | 概要 | 対象ロール |
|--------|--------|------|-----------|
| F-001 | ログイン・ログアウト | メールアドレスとパスワードによる認証 | 全ユーザー |
| F-002 | 日報一覧表示 | 日報の検索・一覧表示 | 全ユーザー |
| F-003 | 日報作成 | 訪問記録・Problem・Planを含む日報の新規作成 | 全ユーザー（自分の日報のみ） |
| F-004 | 日報編集 | 既存日報の内容変更 | 全ユーザー（自分の日報のみ） |
| F-005 | 日報削除 | 既存日報の削除 | 全ユーザー（自分の日報のみ） |
| F-006 | 日報詳細表示 | 日報の閲覧（訪問記録・Problem・Plan・コメント） | 全ユーザー |
| F-007 | コメント投稿 | 日報に対するコメントの投稿 | 全ユーザー |
| F-008 | 顧客マスタ管理 | 顧客の登録・編集・一覧表示 | 上長のみ |
| F-009 | 営業マスタ管理 | 営業担当者の登録・編集・一覧表示 | 上長のみ |

### 3.2 機能詳細

#### F-001: ログイン・ログアウト

- メールアドレスとパスワードでシステムにログインする
- ログイン成功時にJWTトークンを発行する（有効期限: 24時間）
- ログアウト時にトークンを無効化し、ログイン画面へ遷移する
- 無効化されたユーザー（`is_active = false`）はログイン不可とする

#### F-002: 日報一覧表示

- ログイン後のホーム画面として日報の一覧を表示する
- 初期表示時、ログインユーザーの直近1週間の日報を表示する
- 以下の条件で検索可能とする
  - 報告日（From〜To）
  - 営業担当者（プルダウン選択）
- 一覧には報告日・営業担当者名・訪問件数・コメント数を表示する
- 報告日の降順でソートし、1ページ20件のページネーションを設ける
- 一般ユーザーは自分の日報のみ閲覧可能、上長は全営業担当者の日報を閲覧可能

#### F-003: 日報作成

- 営業担当者は1日につき1件の日報を作成できる（同一営業×同一日付のユニーク制約）
- 日報の構成要素は以下の通り

| 項目 | 内容 | 必須 | 制約 |
|------|------|------|------|
| 報告日 | 日報の対象日 | ○ | 当日以前の日付。未来日は不可 |
| 営業担当者 | ログインユーザーを自動設定 | ○ | 変更不可 |
| 訪問記録 | 訪問先の顧客と訪問内容 | ○ | 1件以上、最大20件 |
| Problem | 課題・相談事項 | - | 最大3000文字 |
| Plan | 明日やること | - | 最大3000文字 |

- 訪問記録の詳細

| 項目 | 内容 | 必須 | 制約 |
|------|------|------|------|
| 顧客 | 顧客マスタから選択 | ○ | 有効な顧客のみ選択可能 |
| 訪問内容 | 訪問の内容を自由記述 | ○ | 最大2000文字 |
| 表示順 | 訪問記録の並び順 | ○ | 自動採番 |

- 訪問記録は画面上で動的に行の追加・削除が可能
- Problem・Planは日報1件につきそれぞれ1つのテキスト項目として管理する

#### F-004: 日報編集

- 日報作成者本人のみ編集可能
- 報告日は変更不可とする
- 訪問記録は洗い替え方式（全削除→全挿入）で更新する
- 編集中にキャンセルした場合、確認ダイアログを表示する

#### F-005: 日報削除

- 日報作成者本人のみ削除可能
- 削除時に関連する訪問記録・コメントも同時に削除する（カスケード削除）

#### F-006: 日報詳細表示

- 日報の全内容（訪問記録・Problem・Plan・コメント一覧）を表示する
- 編集ボタンは日報作成者本人のみに表示する
- コメントは作成日時の昇順（時系列順）で表示する

#### F-007: コメント投稿

- 日報の閲覧権限を持つ全ユーザーがコメントを投稿可能
- 上長だけでなく営業担当者本人も返信としてコメント可能
- コメントはフラット構造（スレッドのネストなし）で時系列に並べる
- コメント本文は最大1000文字

| 項目 | 内容 | 必須 | 制約 |
|------|------|------|------|
| コメント本文 | 自由記述 | ○ | 最大1000文字 |
| 投稿者 | ログインユーザーを自動設定 | ○ | 変更不可 |
| 投稿日時 | 投稿時刻を自動記録 | ○ | 変更不可 |

#### F-008: 顧客マスタ管理

- 上長（管理者）のみ操作可能
- 顧客の新規登録・編集を行う（物理削除はなし。有効フラグによる論理削除）
- 無効化した顧客は日報作成時のプルダウンに表示されない
- 無効化した顧客が紐づく過去の訪問記録は、そのまま正しく表示される

| 項目 | 内容 | 必須 | 制約 |
|------|------|------|------|
| 顧客名 | 法人名・個人名 | ○ | 最大100文字 |
| 住所 | 所在地 | - | 最大200文字 |
| 電話番号 | 連絡先 | - | ハイフンあり許容 |
| 担当者名 | 顧客側の窓口担当者 | - | 最大50文字 |
| 有効フラグ | 有効/無効 | - | デフォルト: 有効 |

#### F-009: 営業マスタ管理

- 上長（管理者）のみ操作可能
- 営業担当者の新規登録・編集を行う（物理削除はなし。有効フラグによる論理削除）
- メールアドレスはログインIDとして使用するため、ユニーク制約を設ける
- 無効化されたユーザーはログイン不可とする

| 項目 | 内容 | 必須 | 制約 |
|------|------|------|------|
| 氏名 | フルネーム | ○ | 最大50文字 |
| 部署 | 所属部署 | ○ | 固定値リスト（営業1課/営業2課等） |
| 役職 | 担当/主任/係長/課長/部長 | ○ | 固定値リスト |
| メールアドレス | ログインID | ○ | ユニーク制約。メール形式バリデーション |
| パスワード | ログインパスワード | ○※ | 8文字以上。編集時は空欄で変更なし |
| 有効フラグ | 有効/無効 | - | デフォルト: 有効 |

## 4. 非機能要件

### 4.1 性能要件

| 項目 | 要件 |
|------|------|
| 一覧画面の初期表示 | 3秒以内（日報1000件規模） |
| 日報保存のレスポンス | 2秒以内 |
| 同時接続ユーザー数 | 50ユーザー程度を想定 |

### 4.2 セキュリティ要件

| 項目 | 要件 |
|------|------|
| 認証方式 | JWT（Bearer Token）。有効期限24時間 |
| パスワード保存 | ハッシュ化して保存（平文保存禁止） |
| 通信 | HTTPS必須 |
| 権限制御 | API・画面の両方でロールベースのアクセス制御を実施 |
| セッション管理 | トークン期限切れ時はログイン画面へリダイレクト |

### 4.3 可用性要件

| 項目 | 要件 |
|------|------|
| 稼働時間 | 平日 8:00〜22:00（メンテナンス時間を除く） |
| バックアップ | 日次でデータベースのフルバックアップを取得 |

### 4.4 拡張性に関する備考

本バージョンでは以下の機能はスコープ外とするが、将来的な拡張を考慮してデータ設計を行う。

| 将来機能 | 拡張方針 |
|---------|---------|
| 承認ワークフロー（提出→承認/差戻し） | `daily_reports` テーブルにステータスカラムを追加 |
| Problem/Planの複数行化 | 別テーブルへの分離 |
| コメントのスレッド（ネスト返信） | `comments` テーブルに `parent_comment_id` カラムを追加 |
| 訪問種別（初回/定期/クレーム対応等） | `visit_records` テーブルに種別カラムを追加 |
| 案件・商談との紐付け | 案件テーブルを新設し、訪問記録からFK参照 |
| 添付ファイル | 添付ファイルテーブルを新設 |

## 5. データ設計

### 5.1 エンティティ一覧

| エンティティ | テーブル名 | 概要 |
|------------|-----------|------|
| 営業担当者 | sales_persons | 営業担当者のマスタ情報。ログインアカウントを兼ねる |

### 5.3 主要な制約

| 制約 | 対象 | 内容 |

## 11. 前提条件・制約条件

### 11.1 前提条件

| No. | 内容 |
|-----|------|
| 1 | 利用者は社内ユーザーに限定する（外部公開なし） |
| 2 | 営業担当者・顧客のマスタデータは上長が事前に登録する |
| 3 | 1営業担当者につき1日1件の日報とする |
| 4 | パスワードの初期設定は上長が営業マスタ登録時に行う |

### 11.2 制約条件

| No. | 内容 |
|-----|------|
| 1 | 同時接続ユーザー数は50程度を想定し、大規模利用は対象外 |
| 2 | データベースの文字コードはUTF-8とする |
| 3 | マスタデータの物理削除は行わない（論理削除のみ） |

## 12. 用語集

| 用語 | 説明 |
|------|------|
| 日報 | 営業担当者が1日の活動内容を報告するドキュメント |
| 訪問記録 | 日報の明細行。訪問先の顧客と訪問内容のセット |
| Problem | 日報に記載する課題・相談事項 |
| Plan | 日報に記載する翌日の予定・やること |
| 上長 | 係長以上の役職者。部下の日報閲覧・コメント・マスタ管理を行う |
| 一般ユーザー | 担当・主任の営業担当者。自分の日報のCRUDを行う |
| 論理削除 | データを物理的に削除せず、有効フラグをOFFにすることで無効化する方式 |
| 洗い替え | 更新時に既存の明細データを全削除した後、新しいデータを全挿入する方式 |
| JWT | JSON Web Token。認証トークンの標準形式 |
